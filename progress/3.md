# CatRewrite Progress Report 3: Mathematical Core Roadmap

## Current Status

CatRewrite has reached a useful transition point.

The project is no longer just a textual categorical calculator, and it is no longer just a diagram sketching tool. It now has a working loop:

```text
typed context
-> typed goal
-> generated diagram
-> diagram subgoals
-> tactics
-> pasting completion
-> proof trace
```

The central design invariant remains:

```text
Vue is for interaction.
TypeScript core is for mathematics.
Lean is optional verification later.
```

The next major phase should strengthen the mathematical language in `packages/core`. UI improvements can continue, but the current priority should be the expressive power of the core: common categorical constructions, their universal morphisms, their equations, and eventually parameterized theorem schemas.

Latest verified commands:

```text
npm.cmd run test
npm.cmd run build
```

Current core test status:

```text
54 passing tests across 10 files
```

## What Has Been Built So Far

### 1. Typed Categorical Core

The core supports:

- categories;
- objects;
- functors;
- functor application to objects;
- morphism variables;
- identity morphisms;
- composition;
- functorial action on morphisms;
- hom-types;
- equations;
- contexts;
- structural equality;
- type inference;
- equation typechecking.

The core syntax is intentionally small, but it already supports nontrivial categorical calculations.

### 2. Normalization

The normalizer handles:

- identity laws;
- right association of composition;
- functor preserves identity;
- functor preserves composition.

This lets terms such as:

```text
F.map(eta(adj, X) >> G.map(f))
```

normalize into:

```text
F.map(eta(adj, X)) >> F.map(G.map(f))
```

Normalization is currently definitional simplification. It is not yet a general congruence closure engine.

### 3. Rewrite Engine

The rewrite layer supports:

- rewrite rules;
- rule tags;
- object pattern variables;
- term pattern variables;
- pattern matching;
- RHS instantiation;
- recursive single-step rewriting;
- repeated simplification to a fixed point;
- cycle detection;
- bounded simplification.

This is the backbone for adjunction rules, naturality rules, proved diagram subgoals, and tactics.

### 4. Adjunctions

The core supports:

- `AdjunctionExpr`;
- `AdjunctionDecl`;
- unit terms:

```text
eta(adj, X) : X -> G(F X)
```

- counit terms:

```text
eps(adj, Y) : F(G Y) -> Y
```

- triangle identity rules;
- unit naturality rules;
- counit naturality rules;
- transpose and untranspose helpers.

The first adjunction example is proved:

```text
F.map(eta(adj, X) >> G.map(f)) >> eps(adj, Y) = f
```

The proof uses functoriality normalization, counit naturality, and a triangle identity.

### 5. Natural Transformations

The core supports first-class natural transformations:

- `NatTransExpr`;
- `NatTransDecl`;
- parser syntax:

```text
nattrans alpha : F => G
```

- component terms:

```text
alpha_X : F X -> G X
```

- typechecking for components;
- equality and pretty-printing;
- LaTeX rendering;
- rewrite pattern support for components;
- generated naturality rules:

```text
F.map(f) >> alpha_Y = alpha_X >> G.map(f)
```

There is also tactic support for:

```text
naturality alpha at f
```

The `try` tactic now also considers naturality rules, so ordinary naturality squares can be closed automatically.

### 6. Parser, Pretty Printer, and LaTeX Rendering

The parser currently understands declarations such as:

```text
category C
object X : C
functor F : C -> D
nattrans alpha : F => G
adjunction adj : F -| G
morphism f : X -> Y
```

Term syntax includes:

```text
id(X)
f >> g
F.map(f)
eta(adj, X)
eps(adj, Y)
alpha_X
```

The app now displays most mathematical expressions using KaTeX, including equations in panels and labels inside the diagram.

Composition is displayed in a more conventional mathematical order using `\circ`, while the internal parser and proof engine still use `>>`.

### 7. Tactic Engine

Implemented tactics:

```text
normalize
simp
try
rw <rule>
naturality alpha at f
```

The proof state tracks:

- goals;
- active goal;
- goal status;
- proof steps;
- proof log.

Recent improvements:

- tactics cannot be run again on already proved goals;
- proved subgoals cannot be reproved repeatedly;
- `try` uses both `simp` and `naturality` rules;
- trace messages now mention which rules `try` and `simp` used.

Example trace message:

```text
try closed the goal using alpha.naturality.f
```

For the adjunction example:

```text
try closed the goal using adj.counitNaturality.f, adj.triangle.left.tail.X.f
```

The core has formal diagram data:


Diagrams are not merely visual. Nodes and arrows carry typed mathematical data.

## Current Mathematical Limitations

The mathematical core is still narrow. It is good at:

- basic category syntax;
- functors;
- natural transformations;
- adjunction unit/counit reasoning;
- diagrammatic naturality and pasting examples.

It does not yet support many standard categorical constructions:

- products;
- terminal objects;
- equalizers;
- pullbacks;
- general limits;
- coproducts;
- coequalizers;
- pushouts;
- general colimits;
- monoidal categories;
- cartesian monoidal structure;
- exponentials;
- internal homs;
- algebraic structures internal to a category;
- first-order theorem schemas over arbitrary objects or morphisms.

The next phase should therefore focus on expanding the object language and rule-generation machinery.

## Long-Term Mathematical Plan

The plan below is intentionally layered. Each layer should add real mathematical expressivity while preserving the current architecture:

```text
declaration
-> typed generated objects/terms
-> universal property rules
-> tactics/inspection
-> diagram support
```

## Phase A: Infrastructure for Generated Structure

Before adding many constructions, the core needs a common mechanism for declarations that generate objects, morphisms, and rules.

### A1. Structured Declarations

Current declarations mostly introduce one thing:

```text
object X : C
morphism f : X -> Y
functor F : C -> D
nattrans alpha : F => G
adjunction adj : F -| G
```

Products, equalizers, and limits are different: one declaration generates several things.

Example:

```text
product P of X Y
```

should generate:

```text
object P
morphism pi1_P : P -> X
morphism pi2_P : P -> Y
pairing operation <f,g> : Z -> P
projection rules
possibly an extensionality rule
```

So the core should introduce a general pattern:

```ts
GeneratedStructureDecl
GeneratedTerm
GeneratedRule
```

or specialized declarations that share helper APIs.

### A2. Named Operations vs Synthesized Terms

We need a design decision for operations like pairing.

Possible syntax:

```text
pair(P, f, g)
<f, g>_P
```

Internally it should be a term constructor, not just a named morphism variable, because the typechecker must infer:

```text
pair(P, f, g) : Z -> P
```

from:

```text
f : Z -> X
g : Z -> Y
P = X x Y
```

This suggests adding new `Term` variants for universal constructions, not only adding generated morphism declarations.

### A3. Rule Generation by Structure

There should be a single API that collects generated rules:

```ts
generatedRules(ctx, extraRules)
```

It already collects:

- adjunction rules;
- naturality rules;
- diagram subgoal rules.

It should later also collect:

- product projection rules;
- product eta/extensionality rules;
- equalizer equations;
- equalizer factorization rules;
- pullback commutativity and pairing rules;
- limit cone projection rules.

The rule tags should become more expressive:

```text
simp
naturality
adjunction
triangle
product
limit
universal
extensionality
user
```

## Phase B: Finite Products

Products should be the first new categorical construction.

They are common, simple enough to implement, and useful for later examples involving canonical maps.

### B1. Binary Product Declaration

Candidate syntax:

```text
product P of X Y
```

or:

```text
product P : X * Y
```

The first syntax is probably easier for the current parser.

The declaration should mean:

```text
P is a chosen binary product of X and Y
```

with generated projections:

```text
pi1(P) : P -> X
pi2(P) : P -> Y
```

or named concrete terms:

```text
pi1_P
pi2_P
```

### B2. Pairing Term

Add a term constructor:

```text
pair(P, f, g)
```

with type:

```text
pair(P, f, g) : Z -> P
```

when:

```text
f : Z -> X
g : Z -> Y
P = product(X, Y)
```

### B3. Product Rules

Projection rules:

```text
pair(P, f, g) >> pi1(P) = f
pair(P, f, g) >> pi2(P) = g
```

Eta/extensionality:

```text
h = pair(P, h >> pi1(P), h >> pi2(P))
```

This should probably not be tagged as plain `simp` in both directions, because it can expand terms. It should be used by a dedicated tactic:

```text
product_ext P
```

or by a controlled proof search.

### B4. Product Diagram Support

Products naturally create cones:

```text
      Z
     / \
    f   g
   /     \
  X       Y
```

and the mediating arrow:

```text
Z -> P
```

The diagram layer should eventually understand the projection triangle:

```text
Z --pair(f,g)--> P --pi1--> X
Z ----------------f------> X
```

and similarly for `pi2`.

## Phase C: Terminal Objects

Terminal objects are the nullary case of products and should be implemented close to products.

Candidate syntax:

```text
terminal One : C
```

Generated operation:

```text
terminalMap(One, X) : X -> One
```

Rule:

```text
any two maps X -> One are equal
```

This is more naturally an extensionality/uniqueness principle than a rewrite rule.

Terminal objects are also useful for interpreting global elements:

```text
One -> X
```

## Phase D: Equalizers

Equalizers are the next useful finite-limit construction.

### D1. Equalizer Declaration

Candidate syntax:

```text
equalizer E of f g
```

where:

```text
f, g : X -> Y
```

Generated inclusion:

```text
eqIn(E) : E -> X
```

### D2. Equalizer Equation

Generated rule/equation:

```text
eqIn(E) >> f = eqIn(E) >> g
```

This can be available as a rewrite rule in controlled situations.

### D3. Equalizer Factorization

If:

```text
h : Z -> X
h >> f = h >> g
```

then there is a unique:

```text
factor(E, h, proof) : Z -> E
```

with:

```text
factor(E, h, proof) >> eqIn(E) = h
```

The current core does not have proof objects as term arguments. The first implementation may use a simplified operation:

```text
factor(E, h)
```

and require the equality as a separate subgoal or side condition.

This is our first encounter with dependent side conditions.

## Phase E: Pullbacks

Pullbacks are more diagrammatically important than equalizers for many workflows.

### E1. Pullback Declaration

Candidate syntax:

```text
pullback P of f g
```

where:

```text
f : X -> Z
g : Y -> Z
```

Generated projections:

```text
pb1(P) : P -> X
pb2(P) : P -> Y
```

Generated commutativity:

```text
pb1(P) >> f = pb2(P) >> g
```

### E2. Pullback Pairing

If:

```text
a : W -> X
b : W -> Y
a >> f = b >> g
```

then:

```text
pbPair(P, a, b) : W -> P
```

with:

```text
pbPair(P, a, b) >> pb1(P) = a
pbPair(P, a, b) >> pb2(P) = b
```

Again, side conditions are required. Initially, we can model this as:

```text
pbPair(P, a, b)
```

and let the proof workflow create the commutativity condition as a subgoal.

### E3. Pullback Diagrams

Pullbacks align very well with the existing diagram-subgoal system. The tool should eventually recognize:

```text
P -> X
|    |
v    v
Y -> Z
```

as a square with a distinguished universal property.

This is a strong reason to develop pullbacks soon after products.

## Phase F: General Finite Limits

General finite limits should not be implemented before products, terminal objects, equalizers, and pullbacks.

However, the eventual architecture should point toward:

```text
diagram shape J
functor D : J -> C
limit L of D
projection pi_j : L -> D(j)
cone map mediating(...)
```

This requires:

- small diagram shapes as data;
- objects and morphisms inside an index category;
- cones;
- cone morphisms;
- universal property of terminal object in cone category.

This is more abstract than the current core. We should first implement special cases and then factor common machinery.

## Phase G: Colimits

After finite limits:

- coproducts;
- initial objects;
- coequalizers;
- pushouts;
- general finite colimits.

The dual constructions should reuse as much machinery as possible, but not be forced too early. In practice, products and pullbacks are more urgent for the user's current direction.

## Phase H: Cartesian and Monoidal Structure

Once products exist, we can add a cartesian monoidal view:

```text
X x Y
1
diagonal delta_X : X -> X x X
terminal map !_X : X -> 1
```

Rules:

```text
delta_X >> pi1 = id(X)
delta_X >> pi2 = id(X)
```

and product functoriality:

```text
f x g : X x Y -> X' x Y'
```

This becomes important for Frobenius-style canonical morphisms.

## Phase I: First-Order / Schema-Level Constructions

The user explicitly wants support for statements such as:

```text
for every object P, the canonical Frobenius morphism gamma_P is an isomorphism
```

where `gamma_P` may be a canonical expression built from projections, pairings, functorial images, diagonals, or product/limit structure.

This is a major expressive jump. It requires a layer above individual contexts.

### I1. Object Variables in Theorem Schemas

Currently the context contains concrete declared objects:

```text
object X : C
object Y : C
```

Schema-level mathematics needs object variables:

```text
forall object P : C, ...
```

or declaration syntax:

```text
schema frobenius(P : C) : ...
```

The core already has object pattern variables for rewrite matching, but those are not the same as first-order quantified variables in user-facing theorems.

We need:

- schema variables;
- instantiation;
- side conditions;
- generated theorem instances.

### I2. Canonical Morphism Builders

For a theorem like:

```text
gamma_P = <F(pi1), F(pi2), ...>
```

the system must construct a term from structural data.

This suggests adding canonical-builder APIs:

```ts
canonicalProductMap(...)
canonicalComparisonMap(...)
canonicalFrobeniusMap(...)
```

The user should not have to manually spell every projection and pairing if the construction is canonical.

### I3. Isomorphism as First-Class Structure

To state:

```text
gamma_P is an isomorphism
```

we need either:

```text
iso gamma_P
```

or a declaration:

```text
isomorphism gamma_P with inverse delta_P
```

Mathematically, this means adding:

- `IsoExpr`;
- inverse morphism terms;
- equations:

```text
gamma_P >> gamma_P.inv = id(...)
gamma_P.inv >> gamma_P = id(...)
```

The proof state must support goals that are not only equations, or encode isomorphism goals as pairs of equations.

The simpler first implementation should encode:

```text
isIso(gamma)
```

as two generated equation subgoals once an inverse candidate is provided.

### I4. Frobenius-Style Canonical Morphisms

The user's example points toward constructions common in categorical semantics:

```text
gamma_P = < F(pi_1), F(pi_2), ... >
```

The exact form depends on the surrounding structure. It may involve:

- products in the source category;
- products in the target category;
- a functor `F`;
- comparison maps;
- pairing;
- projections;
- possibly preservation of products or Frobenius reciprocity conditions.

The core should not hardcode one Frobenius map too early. Instead, implement the ingredients:

1. products;
2. product pairings;
3. functor action on products;
4. comparison maps;
5. isomorphism goals;
6. theorem schemas over arbitrary objects;
7. canonical construction macros.

Then a Frobenius theorem can be expressed as a schema:

```text
schema frobeniusGamma(P : C):
  gamma(P) : ...
  inverse gammaInv(P) : ...
  prove gamma(P) >> gammaInv(P) = id(...)
  prove gammaInv(P) >> gamma(P) = id(...)
```

### I5. Side Conditions and Assumptions

Many first-order constructions require assumptions:

```text
for all P, if P has ...
for any f, if square commutes, then ...
assuming F preserves products
```

This requires assumption declarations and local hypotheses.

Possible syntax:

```text
assume h : f >> g = f >> k
```

or schema syntax:

```text
forall object P : C
assume product PX of ...
prove ...
```

The current core has `EquationDecl`, but equation declarations are not yet used as local hypotheses in a mature way. This should be strengthened.

## Phase J: Proof Search and Automation

As the mathematical language grows, `try` needs to evolve.

Current `try`:

```text
normalize
-> simplify with simp/naturality
-> normalize
-> simplify again
-> compare sides
```

Future `try` should become layered:

```text
try0: normalize only
try1: simp/naturality
try2: product projection simplification
try3: extensionality when goal target is product
try4: diagram subgoal/local rules
try5: bounded search over applicable rules
```

The important point is that proof search must remain explainable. Every successful automatic proof should report:

- which rules were used;
- which extensionality principles were invoked;
- which generated construction rules were used;
- which subgoals were created or solved.

## Recommended Implementation Sequence

### Immediate Next Milestone: Binary Products

The next concrete mathematical task should be:

```text
binary products in the core
```

Implement in order:

1. `ProductDecl` in syntax.
2. Parser syntax:

```text
product P of X Y
```

3. Product projection terms:

```text
pi1(P)
pi2(P)
```

4. Product pairing term:

```text
pair(P, f, g)
```

5. Typechecking:

```text
pi1(P) : P -> X
pi2(P) : P -> Y
pair(P, f, g) : Z -> P
```

6. Pretty and LaTeX rendering.
7. Rewrite rules:

```text
pair(P, f, g) >> pi1(P) -> f
pair(P, f, g) >> pi2(P) -> g
```

8. Tests proving projection equations by `try`.
9. UI rule browser display for product rules.

Do not start with product extensionality as an unrestricted simp rule. Add it later as a tactic.

### Second Milestone: Product Extensionality

Add:

```text
product_ext P
```

for goals between maps into a product.

If the goal is:

```text
f = g : Z -> P
```

where `P` is a product, generate or prove subgoals:

```text
f >> pi1(P) = g >> pi1(P)
f >> pi2(P) = g >> pi2(P)
```

This integrates naturally with the existing diagram subgoal workflow.

### Third Milestone: Terminal Objects

Add terminal objects as nullary products.

### Fourth Milestone: Pullbacks

Pullbacks should come before equalizers if diagrammatic usefulness is the priority.

They create square-shaped reasoning that fits CatRewrite's visual proof mode very well.

### Fifth Milestone: Equalizers

Add equalizers after pullbacks or before them depending on which examples become urgent.

Equalizers will force us to handle side conditions more explicitly.

### Sixth Milestone: Isomorphism Goals

Add first-class support for:

```text
isIso(f)
```

or a structured `IsoDecl`.

At first, expand an isomorphism goal into two equation subgoals with a user-provided inverse.

### Seventh Milestone: Schema Layer

Only after products, pullbacks, and isomorphism goals are stable should we add first-order theorem schemas.

This is where statements like:

```text
for every object P, gamma_P is an isomorphism
```

become natural.

## Strategic Principle

The mathematical core should grow by making universal properties executable.

For each construction, the implementation should answer:

1. What new declarations does the user write?
2. What objects and morphisms are generated?
3. What equations are generated?
4. Which equations are safe simplification rules?
5. Which principles require a tactic rather than simp?
6. How does the diagram display the construction?
7. How does the proof trace explain the construction?

This principle will keep CatRewrite from becoming only a bag of rewrite rules. The goal is a diagrammatic categorical assistant whose automation reflects the actual mathematical structure.

## Summary

CatRewrite currently has:

- a typed categorical syntax;
- normalization;
- rewriting;
- adjunctions;
- natural transformations;
- parser and pretty printer;
- KaTeX rendering;
- tactics with rule-aware trace messages;
- diagram generation from goals;
- auxiliary nodes and arrows;
- diagram regions;
- diagram subgoals;
- proved-region marking;
- pasting completion;
- proof-state UI.

The next phase should focus on:

```text
products -> terminal objects -> pullbacks/equalizers -> finite limits
```

and then:

```text
isomorphisms -> theorem schemas -> canonical first-order constructions
```

The long-term target is to support mathematical statements of the form:

```text
for every object P, the canonical morphism gamma_P is an isomorphism
```

not as ad hoc text, but as typed, generated, checkable categorical structure.
