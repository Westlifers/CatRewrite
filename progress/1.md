# CatRewrite Progress Report and Forward Plan

## 1. Current Status

CatRewrite has moved from the initial project setup into a working local prototype with a typed categorical core, a textual proof workflow, and an early diagram interaction layer.

The project is currently an npm workspace with:

```text
packages/core
packages/app
```

The most important architectural invariant from the original plan is being followed:

```text
Vue is for interaction.
TypeScript core is for mathematics.
Lean is optional verification later.
```

All mathematical operations implemented so far live in `packages/core`. The Vue app calls those core APIs and does not contain proof-kernel logic.

## 4. Diagram Work Completed So Far

Implemented:


A diagram is stored as formal data, not as a picture.

Each node has:

- id;
- label;
- typed object;
- position.

Each arrow has:

- id;
- source node;
- target node;
- label;
- typed term;
- status.

### Path-to-Equation Conversion

Implemented:

- `pathTerm`;
- `regionEquation`.

Given two explicit paths, the core composes arrow terms and typechecks the resulting equation.

### Current Goal Diagram

Important correction completed:

The diagram no longer depicts a fixed counit naturality square in the app.

The app now generates a diagram from the current goal equation:

```text

```

For the transpose example, the diagram represents the normalized goal:


The upper lane is the left-hand path. The lower lane is the right-hand path.

## 6. Important Design Corrections Made During Development

### Correction 1: npm Instead of pnpm

The plan suggested pnpm. The local environment did not have pnpm available, so the project uses npm workspaces.

### Correction 2: Diagram Should Depict the Goal

An early diagram prototype showed the counit naturality diagram directly. This was misleading because the diagram should represent the current goal.

This has been corrected.

The app now generates the diagram from the current goal equation.

### Correction 3: Path Selection Should Be Click-Based

Manual left/right path text fields were too fragile. The UI now supports arrow-click path selection.

### Correction 4: Vue Flow Was Not the Right Diagram Renderer

Vue Flow was removed from the visible diagram rendering path. A custom SVG renderer gives better control for categorical diagrams.

## 7. Current Limitations

### 7.1 Diagram Generation Is Still Simple

The current equation-to-diagram generator lays out a normalized equation as two lanes:

```text
left side path
right side path
```

It is useful, but it is not yet a sophisticated categorical diagram layout engine.

Limitations:

- no automatic square/triangle recognition;
- no region inference;
- no layout optimization for larger diagrams;
- no support yet for multiple shared intermediate objects across paths beyond basic node reuse.

### 7.2 Auxiliary Arrow UI Is Still Text-Based

The tool can verify an auxiliary arrow from a term, but the user still enters source and target node ids manually.

The intended later behavior is:

```text
click source node
click target node
enter term
tool verifies and adds arrow
```

### 7.3 Subgoals Are Not Yet Diagram-Proof-State Objects

The app can make a goal from selected paths, but the real desired workflow is:

```text
selected paths -> subgoal
subgoal proved -> marked region
outer goal reduced by proved regions
```

This is not finished yet.

### 7.4 No Pasting/Reduction Yet

The tool does not yet formally reduce an outer diagram after a subdiagram has been proved.

This is a major upcoming step.

### 7.5 Tactics Are Useful but Still Early

Current tactics prove the basic examples, but:

- `try` is not yet robust bounded search;
- `rw` is basic;
- tactic traces are not yet full human-readable calculation chains;
- no advanced mate tactic yet.

### 7.6 No LaTeX or Lean Export Yet

MVP 9 and MVP 10 are not started.

## 8. Recommended Next Steps

The next work should continue the diagram-proof-state path, step by step.

## 8.1 Next Step: Diagram Subgoals

Add a formal diagram proof state:

```ts
interface DiagramProofState {
  diagram: Diagram;
  outerGoalRegionId: string;
  subgoals: DiagramSubgoal[];
  provedRegionIds: string[];
}
```

Add:

```ts
createSubgoalFromPaths(...)
```

The user should be able to:

1. select arrows for a left path;
2. select arrows for a right path;
3. click `Add Subgoal`;
4. see the subgoal appear in a list.

The subgoal should not replace the main goal.

## 8.2 Prove Subgoals

Each diagram subgoal should be connected to the existing tactic engine.

Subgoal list should show:

- equation text;
- hom-type;
- status;
- selected tactic;
- proof button;
- proof trace.

When a subgoal is proved, mark it as proved.

At first, marking can be only in the subgoal list. Later, highlight the corresponding region/path in the diagram.

## 8.3 Node-Based Auxiliary Arrow UI

Replace source/target id text fields with direct diagram selection.

Suggested interaction:

1. Click `Add Auxiliary`;
2. click source node;
3. click target node;
4. enter term;
5. tool verifies term hom-type;
6. arrow appears.

Keep text fallback for debugging.

## 8.4 Use Proved Subgoals as Rewrite Rules

Once a subgoal is proved, the equation should become available as a local rewrite rule.

For example, a proved naturality square:

```text
F.map(G.map(f)) >> eps(adj, Y)
=
eps(adj, F X) >> f
```

should be usable inside the outer goal.

This is the bridge between diagram interaction and tactic rewriting.

## 8.5 Diagram Reduction / Pasting

After subgoals can be proved and used as local rules, implement simple diagram reduction:

```text
outer path contains proved subpath
replace proved subpath by equivalent path
update remaining goal
```

This should be formal, not visual-only.

The first version can operate directly on path arrays and terms.

## 8.6 Better Diagram Layout

Improve layout for:

- long paths;
- repeated objects;
- triangles;
- squares;
- auxiliary arrows;
- proved subregions.

This should remain custom SVG unless a strong reason appears to add a layout package.

## 8.7 LaTeX Export

After the proof trace and diagram state are more meaningful, add LaTeX export:

- aligned equation calculation;
- `CD` environment diagram;
- context summary.

The user explicitly prefers `CD` over `tikzcd`.

## 8.8 Lean Export

After core examples and diagram proof traces stabilize, add Lean skeleton export:

```lean
import Mathlib.CategoryTheory.Adjunction.Basic

theorem example : ... := by
  sorry
```

Full Lean verification should remain later.

## 9. Short-Term Implementation Sequence

Recommended immediate sequence:

1. Add `DiagramProofState`.
2. Add `DiagramSubgoal`.
3. Add `Add Subgoal` from selected paths.
4. Add subgoal list UI.
5. Add `Prove Subgoal`.
6. Convert proved subgoals into local rewrite rules.
7. Use local proved subgoals to simplify the outer goal.
8. Improve auxiliary-arrow source/target selection by clicking nodes.
9. Add visual marking for proved arrows/regions.

This keeps the project aligned with the original plan while avoiding a premature jump into complex automatic region detection.
